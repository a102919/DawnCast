# DawnCast 使用者流程分析（中期評估）

> 分析基礎：5 位受訪者（p1 林佳穎 / p2 陳冠廷 / p3 黃詩涵 / p4 蘇怡君 / p5 Kevin）深度訪談 + 結構化問卷，MVP 前端 mock data 階段，pre-launch，無真實付費用戶。
> 紅線：標出現有 UI 的斷點，不設計新功能、不發明像素，所有 mitigation 都在現有架構內可實作。

---

## 一、資訊架構檢視（Information Architecture Review）

### 1.1 現有底部導覽（currentNav）

| Tab | 圖示 | 路由 | 用途 |
|-----|------|------|------|
| 1. 首頁 | Home | `/` | 集數卡列表 + Hero CTA |
| 2. 單字本 | BookMarked | `/vocab` | 收藏列表 + 搜尋 + 閃卡入口（按鈕在頁底） |
| 3. 升級 / 進度 | ArrowUpCircle / BarChart2 | `/pricing`（Free）／`/progress`（Pro） | 動態切換，Free 看方案頁，Pro 看進度頁 |
| 4. 設定 | Settings | `/settings` | 字型、播放速度預設、主題、訂閱管理 |

> 底部導覽共 4 個固定 tab，播放器（PlayerRoute）無 tab 入口，只能從首頁集數卡點擊進入。

### 1.2 IA 問題

1. **第三 tab 對 Free 用戶是「升級」CTA，對 Pro 用戶切換成「進度」，但 ProgressRoute 內容三個欄位均硬寫 `value="--"`，等同空殼。** 5 位受訪者中無一主動點進進度頁後有正向回饋；Free 用戶的「升級」tab 實際被點擊只發生於查詞被擋後的氣憤式點擊，非主動意願。tab 的雙重身份讓兩類用戶都拿不到實質價值，是最顯著的導覽位置錯置。

2. **設定頁承載「每次播放都用得到」的個人化設定，但被埋在第四 tab 且僅被開啟 1 次/週（5 位受訪者）。** 播放速度預設、字幕字型大小放在 SettingsRoute，p1、p4、p5 三位明確表示「改了好像沒生效或下次要重調」。SettingsContext 持久化邏輯存在，但 PlayerControls.tsx 的速度按鈕以元件內部 local state 呈現、未讀取 settingsContext.playbackRate，導致「改了不生效」是實際 bug 而非使用者誤解。

3. **單字本頁的「開始閃卡複習」按鈕位於頁面最底，與 5 位受訪者的回訪路徑脫節。** 4 位受訪者描述「看到但沒按」「以為是另開新分頁」「第七天才按過一次」；且底部導航 4 個 tab 沒有任何一個直接指向閃卡，要進入閃卡需要兩層點擊（底部 tab → 單字本頁 → 捲到底 → 按鈕），入口可發現性與實際使用意願成反比。

4. **首頁集數卡對鎖定集數採 opacity-75 + grayscale + Lock 圖示，視覺上與「未聽過但可播」難以區分。** p3 黃詩涵原話「掃過去以為都還沒聽過，點錯兩次才學會」；同時鎖頭卡片點擊後的 bottom sheet 僅顯示「此集需要 Pro」，未帶出該集的主題、CEFR、時長，付費前無法判斷 Pro 內容是否值得。

5. **首頁缺少「學習儀表板」這個 IA 層級。** 5 位受訪者均表達「不知道這週學了什麼」「沒有進步感」，目前首頁直接是集數卡列表，沒有任何累積指標的視覺節點。這不是新功能問題，是 IA 上把「我的進度」擠到 Pro 用戶的進度頁（且為空殼），Free 用戶與低開啟頻率的 Pro 用戶完全無從看見自己的學習累積。

6. **查詞 popup（WordCardPanel）以 `fixed` 從底部滑入，實際遮蓋 CueDisplay 字幕區。** popup 高度 max-h-80 在行動裝置約佔 40–50% 螢幕，與字幕區重疊。這不是單一元件問題，是 IA 上「查詞」與「跟讀字幕」兩個核心行為沒有並列位置的設計缺口，p1、p2 均描述「查詞就不能跟讀」。

7. **底部導覽無播放器快速回入口。** p2、p5 明確描述「從單字本切回播放器要先回首頁找那集再點進去，底部導航四個 tab 沒有一個是播放器，每次都要繞一圈」。付費用戶的核心行為（聽 → 查詞 → 查單字本 → 回去聽）的返回路徑完全無捷徑。

## 二、關鍵流程現況分析

### 流程 1：首次收聽（First Listen）

**觸發條件**：點擊首頁無鎖集數卡，進入 PlayerRoute。

**現有步驟（描述現有 UI 實際狀態）**

1. 開啟 app → 直接進入 HomeRoute，無引導 onboarding、無註冊/登入牆（pre-launch 階段免登入）。
2. Hero 區顯示兩顆 CTA：「免費開始聆聽」、「了解訂閱方案」。
3. 下方集數卡列表：最新 1 集可點擊播放，其餘顯示鎖頭 + grayscale 樣式（opacity-75 + grayscale CSS filter）。
4. 點可用集數卡 → Link to `/player` → PlayerRoute 直接播放，無功能說明 tooltip。
5. 播放中 → CueDisplay 英文當前句大字高亮 + 中文小字對照。
6. 點擊 CueLine 任一英文單字 → handleWordClick → lookupCount 累加，WordCardPanel 從底部滑入。
7. 第 3 次查詞後 lookupLimitReached = true → popup 底部顯示升級 soft CTA。
8. 退出播放器 → 重開 → currentTime 回 0（不跨 session 持久化）。

**估計完成率（聽完第一集 80% 以上，觸發 markAsListened）**：65–75%

**斷點與 dropOffRisks**

| 步驟 | 斷點原因 | 緩解（改善現有設計） |
|------|----------|---------------------|
| 步驟 3：大量鎖頭卡片構成視覺壓迫 | Free 用戶首印象是「內容都被鎖了」，p2、p3、p5 描述「視覺壓迫」「陪榜感」，對 Pro 內容質量無從感知，付費動機瓦解 | 鎖頭卡維持鎖頭圖示但保留完整標題 + 主題 + CEFR 標籤（EpisodeCard 已有 props，只需移除 grayscale filter 或降低不透明度而非完全去色），讓用戶掃過首頁時能讀到 Pro 內容的主題訊號 |
| 步驟 4：播放器首次進入無功能提示 | p1「第三天才找到字幕字型調整」、p4「誤打誤撞才點到字幕查詞」；查詞 popup 是最高價值功能卻靠運氣發現 | PlayerRoute 首次進入時在控制欄下方顯示一次性 tooltip「點擊字幕任一英文單字可查詞」，sessionStorage 旗標控制只顯示一次，不阻斷播放 |
| 步驟 6：WordCardPanel 底部固定彈出遮蓋 CueDisplay | 5 位受訪者中 4 位描述「查詞就沒辦法跟讀」「看不到當前句」；popup max-h-80 在手機約佔 40–50% 螢幕 | popup 改為 max-h-48 並補 bg-black/85 backdrop-blur，讓底層字幕在 popup 開啟時仍可讀；drag-to-dismiss 保留 |
| 步驟 7：soft CTA 在學習投入高峰出現 | p1「額度用完當下很賭爛」、p3「當下直接切回 IG」；雖非全屏 modal，但時機在注意力高點破壞投入感 | 查詞限制的升級推薦延後到集末播放完成時（PlayerRoute 已實作 onEnded 鉤子），onEnded 時顯示一次升級卡；查詞當下只改文案為「剩餘 0 次 · 升級後無限查詞」不加 CTA 按鈕 |
| 步驟 8：播放進度不跨 session 持久化 | p1、p2、p4、p5 均遭遇「重開找不到上次位置」，需從頭拖或重新聽；通勤中斷後無法接續是最直接的放棄觸發點 | PlayerContext 在 currentTime 變化時寫入 localStorage[dc-position-${episodeId}]，PlayerRoute mount 時讀取並 seekTo；seekTo 已實作於 PlayerControls，只差持久化層 |

### 流程 2：儲存單字（Save Vocab）

**觸發條件**：在 WordCardPanel 點擊「加入單字本」。

**現有步驟（描述現有 UI 實際狀態）**

1. WordCardPanel 顯示 → 點「加入單字本」→ addVocab 寫入 VocabContext（包含 word、pos、meaning、sourceEpisodeId、sourceTimestamp、sourceLineNo）。
2. 按鈕狀態切為「已收錄」（Check 圖示 + success 色）。
3. 離開播放器 → 點底部「單字本」tab → VocabRoute。
4. 列表顯示：單字 + 詞性 + 釋義 + 來源集數（sourceEpisodeId 顯示為「第 X 集」）。
5. 搜尋欄 onChange 即時過濾（useMemo 依賴 query）。
6. 捲到頁底 → 看到「開始閃卡複習」按鈕 → 點擊無實際跳轉（VocabRoute 內閃卡邏輯為佔位）。

**估計完成率（存入單字本 → 完成至少一次閃卡複習）**：0%（「開始閃卡複習」按鈕為佔位元件，步驟 6 整條流程斷掉）

**斷點與 dropOffRisks**

| 步驟 | 斷點原因 | 緩解（改善現有設計） |
|------|----------|---------------------|
| 步驟 1：addVocab 未存入 sourceSentence | p1、p2、p3、p5 描述「不知當初為何收藏」「跟 Google 翻譯存下來沒兩樣」；WordCardPanel 持有 activeCue.text 但未傳入 addVocab | addVocab 呼叫處將 activeCue.text 一同寫入 VocabContext；VocabEntryCard 補一段小字「來自：『...』」（截斷處理）；現有資料流完整，只缺欄位傳遞 |
| 步驟 4：Free 50 額度到頂無提前警示 | p3「第五天按閃卡才跳錯誤才知道滿了」、p5「被擋住更幹，因為是想複習不是新增」；到頂才報錯是沉沒成本衝突點 | VocabContext 在長度 ≥ 45 時於 VocabRoute 頂部顯示 soft banner「單字本將滿（X/50）· 升級 Pro 解鎖無限」，先於 50 觸發 |
| 步驟 6：「開始閃卡複習」按鈕為佔位，點擊無行為 | 4 位受訪者明確表示「按了沒反應」；整條 Save→Review 閉環在此斷掉，直接讓單字本淪為「只進不出的垃圾桶」 | VocabRoute 補實作基本翻卡邏輯：modal 顯示 vocab.word → 點擊翻面顯示 meaning + sourceSentence + 「認識 / 不認識」二元評分；資料結構 VocabEntry 已具備，缺的是 JSX 與 state |
| 步驟 6（前提）：閃卡按鈕在頁底，需捲動才可見 | 5 位受訪者中 4 位描述「看到但沒按」，入口可發現性低 | 將「開始閃卡複習」從頁底移至頁頂單字數量旁（如「X 個單字 · 開始複習」），只動 VocabRoute JSX 排版 |

### 流程 3：SRS 閃卡複習（SRS Review）

**觸發條件**：在單字本頁點擊「開始閃卡複習」，或從每日提醒進入。

**現有步驟（描述現有 UI 實際狀態）**

1. 點「開始閃卡複習」→ FlashcardRoute（目前按鈕為佔位，此分析以 FlashcardRoute 現有程式碼實際邏輯描述）。
2. FlashcardRoute 從 VocabContext 取出 dueCards（依 SRS nextReview 時間排序），若 dueCards.length === 0 顯示「今天沒有需要複習的單字」。
3. 顯示正面：英文單字大字 + 「翻面看答案」按鈕。
4. 點「翻面」→ 顯示：音標 + 詞性 + 中文釋義 + 例句（sourceLineNo 對應的 cue 句子）。
5. 顯示「認識」/「不認識」兩個按鈕 → updateReviewResult → 計算下次 nextReview。
6. 若 dueCards.length > 10，直接全部排入同一回合，無上限機制，無進度條顯示當前第幾張/共幾張。
7. 所有卡片完成 → 顯示「複習完成」靜態畫面，無成就感反饋，無回到單字本按鈕。

**估計完成率（觸發複習 → 完成當日全部到期卡）**：20–35%（堆積超過 15 張後幾乎無人堅持；p5 堆積至 40+ 張後放棄；唯一完成路徑是卡片少於 5 張時）

**斷點與 dropOffRisks**

| 步驟 | 斷點原因 | 緩解（改善現有設計） |
|------|----------|---------------------|
| 步驟 2：無每日複習上限，任何空窗期都造成卡片堆積 | 全員（5/5）在訂閱第 1.5–3 週放棄閃卡，直接原因是「打開看到一大堆」；p5「到期卡堆到 40 幾張，看到那數字就有壓力，乾脆不開」 | FlashcardRoute 加入每日上限設定（預設 10 張，用戶可在 SettingsRoute 調整 5/10/20），dueCards.slice(0, dailyLimit) 傳入複習回合；SettingsContext 已有擴充空間 |
| 步驟 4：閃卡例句缺少來源集數標題與情境說明 | p1「quarterly review、leverage、synergy 全混在一起，完全不知道哪集的情境」；失去脈絡是複習感覺「沒用」的根因 | 翻面同時顯示 sourceEpisodeId 對應的集數標題（如「來自 Ep.12：AI in the Workplace」），只需從 EPISODES 常數查欄位，不需 API |
| 步驟 5：SRS 計算邏輯對用戶不透明 | p5「它決定今天給我幾張，我不知道為什麼，有被 app 指揮的感覺」；不透明邏輯引發抗拒 | 翻面下方補一行「下次複習：認識 → X 天後 / 不認識 → 明天」，透明化 SRS 邏輯不需改演算法 |
| 步驟 6：無進度條 | 用戶不知道剩幾張，無法預期結束時間，通勤等時間有限情境下直接放棄 | FlashcardRoute 頂部補進度條與計數「3 / 10」，現有 dueCards array 有 index，只差 JSX 呈現 |
| 步驟 7：複習完成無正向反饋 | 完成後看到靜態文字，無成就感；p4「做完了然後呢，就沒了」；比「壓力大」更傷留存的是「做完沒感覺」 | 「複習完成」頁改為顯示「今日複習 X 張，連續 N 天」+ 動態視覺回饋（CSS animation），並補「查看單字本」按鈕；數字從 VocabContext 計算 |

### 流程 4：續訂決策（Renewal Decision）

**觸發條件**：Pro 訂閱接近到期日（模擬），或用戶主動思考「這個月值不值得」。

**現有步驟（描述現有 UI 實際狀態）**

1. 用戶在月底查看手機帳單，或主動打開設定頁確認訂閱狀態。
2. 設定頁 → 「訂閱管理」區塊 → 目前顯示訂閱方案名稱 + 到期日靜態文字。
3. 點「訂閱管理」→ 無跳轉（佔位元件，無法深入）。
4. 用戶嘗試找取消入口 → 找不到 → 放棄取消（如 p1、p4 案例）。
5. 用戶回首頁 → 看到最新集封面主題 → 若主題相關則留下，若無關則取消意圖升溫（如 p2、p5 的「靠運氣留住」）。
6. 在無任何介入的情況下，留存與否由「本週有沒有碰巧出現相關主題的新集」決定。

**估計完成率（主動評估後決定續訂）**：30–45%（實際留存靠懶惰 + 運氣，非產品設計；若用戶認真找取消入口且主題連續無關，流失率預計 60%+）

**斷點與 dropOffRisks**

| 步驟 | 斷點原因 | 緩解（改善現有設計） |
|------|----------|---------------------|
| 步驟 1–2：整個 app 缺乏「本月我學了什麼」的自動回顧 | 5 位受訪者均表達「不知道自己有沒有進步」「說不出 Pro 給我什麼具體值得」；月底看扣款時無法自我說服；p5「這 3 個月我能說的是『我有在聽』，不能說『我有在進步』」 | ProgressRoute（已存在但為空殼）補入三個數字：本月聽了幾分鐘、本月查了幾個詞、本月單字本新增幾個；數據已在 PlayerContext / VocabContext，只差統計與渲染 |
| 步驟 3：「訂閱管理」點進去無內容（佔位） | p1「找半天只看到訂閱管理點進去沒什麼東西」；隱藏取消入口屬假性留存，且傷害品牌信任 | 「訂閱管理」補入：方案名稱 + 下次扣款日期 + 「管理訂閱」（連結到 iOS/Android 系統設定，pre-launch 先補外連）；誠實呈現訂閱狀態 |
| 步驟 5：首頁無主題篩選，「本週有沒有相關集數」完全靠運氣 | p2「靠運氣留住我，不是靠 app 設計」、p5「差那一集，那週沒開我可能就走了」 | 首頁集數卡列表補主題 tag 篩選（情境類別），讓用戶自己找到相關集；EpisodeCard 的 topic 欄位已有資料，只差首頁 filter UI |
| 步驟 6：低頻使用月份無任何介入機制 | p1「一週用不到三次又閃卡停了，在浪費錢」；5 位受訪者的取消門檻都是「使用頻率自我感知下降」而非功能問題 | Pro 用戶連續 5 天未開 app 時，下次開啟首頁 Hero 區顯示一次「歡迎回來 · 你上次聽了 Ep.X，繼續？」banner（PlayerContext 已有 lastPlayedEpisodeId），把低頻用戶導回最近的集數 |
| 步驟 4（意外）：設定頁取消入口找不到，意外留住 p1、p4 | 這是假性留存：用戶留下的原因是「找不到取消鈕」而非「覺得值得」；下個月若對品牌不信任累積，流失會更快且口碑為負 | 誠實呈現訂閱狀態與取消路徑（見步驟 3 改善方向），把資源投在提升真實感知價值 |

## 三、跨流程共同缺口

以下問題不屬單一步驟，需橫向處理：

- **播放速度設定頁與播放器未連動（已確認 bug）**：SettingsContext.playbackRate 持久化層存在，PlayerControls.tsx 的速度按鈕卻以 local state 呈現。p1、p4、p5「改了不生效」「每次重進回到 1x」皆可還原至此根因。修復方式：PlayerControls 移除內部 state，改為 useSettings().playbackRate，setter 同步回 settingsContext。
- **播放進度跨 session 不持久化**：PlayerContext.currentTime 只在記憶體；p1、p4、p5 重開後找不到位置。修復方式見流程 1 步驟 8。
- **ProgressRoute 三個欄位硬寫 value="--"**：即使 Pro 用戶點進進度頁，也只看到三個空白數字；是 IA 上進度功能尚未實作完成，屬最高優先缺口（直接影響流程 4 的續訂決策）。
- **缺乏「已查過單字」識別**：p3、p5 明確表示重複查同字被扣額度的反直覺設計。修復：VocabContext 已有 hasVocab(word) 判斷，若該字已收藏則不計入 lookupCount；只多寫一行條件式，不影響其他邏輯。
- **AB 段落重複功能入口不存在於現有 UI**：p2、p3、p4 三位找不到，等同功能不存在；5 位皆未實際使用過一次。現有播放器控制欄有空間加入 AB 按鈕，此為 Pro 差異化功能中可見性最低的一項。
